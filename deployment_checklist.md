# QA 종료 기준 (Exit Criteria)

## 배경

버즈빌은 스쿼드 형태로 빠르게 실험하고 기능을 개발합니다. 체계적인 QA 종료 기준을 정의하여, 배포 가능 여부를 일관되게 판단하고 불필요한 지연을 방지합니다.

## 기본 원칙

- **유저 영향도**와 **비즈니스 리스크**(금전적 손실, 매체사 영향)를 기준으로 배포 여부를 판단합니다.
- 리스크가 낮은 변경은 빠르게 배포하고, 리스크가 높은 변경은 충분한 검증 후 배포합니다.

---

## 배포 판단 매트릭스

| | 비즈니스 리스크 낮음 | 비즈니스 리스크 높음 |
|---|---|---|
| **유저 영향 낮음** | 즉시 배포 가능 (Dev QA 완료 시) | QA 검증 필수 |
| **유저 영향 높음** | QA 검증 필수 | QA 검증 필수 + 단계적 롤아웃 권장 |

**비즈니스 리스크 높음 예시**: 과금/정산 로직, 리워드 지급, 매체사 연동, 광고 노출/클릭 집계

**유저 영향 높음 예시**: 주요 사용자 플로우 변경, 대규모 UI 변경, 퍼포먼스에 영향을 주는 변경

---

## 기능테스트 종료 기준

기능테스트는 전체 기능이 요구사항대로 동작하는지 검증하는 단계입니다.

### 필수 조건

- [ ] 요구사항에 정의된 기능이 정상 동작함
- [ ] 테스트 케이스 실행 완료 (Pass rate 목표 달성)
- [ ] Urgent 결함 0건
- [ ] High 결함 0건 (또는 잔여 결함 수용 합의 완료)
- [ ] 모든 이슈가 해결 상태 (Staging QA DONE / Prodmini QA DONE / Prod QA DONE)

### 권고 조건

- [ ] Medium 결함 5건 이하 (초과 시 스쿼드 합의 필요)
- [ ] 매체사 영향이 있는 경우, 호환성 검증 완료
- [ ] 미해결 이슈에 대한 후속 처리 계획 수립됨

### 기능테스트 → 리그레션테스트 전환 기준

- [ ] 위 필수 조건 모두 충족
- [ ] 발견된 버그 수정 빌드 배포 완료 (Staging)
- [ ] 수정된 버그에 대한 Retest 완료

---

## 리그레션테스트 종료 기준

리그레션테스트는 버그 수정 및 변경사항이 기존 기능에 영향을 주지 않는지 검증하는 단계입니다.

### 필수 조건

- [ ] 수정된 버그 전체 Retest 완료 (PASS 확인)
- [ ] 핵심 시나리오 리그레션 TC 전체 PASS
- [ ] Urgent / High 결함 0건
- [ ] 수정 과정에서 신규 결함 발생 여부 확인 완료

### 권고 조건

- [ ] 플랫폼별 (iOS / Android) 핵심 플로우 PASS 확인
- [ ] 성능 저하 없음 확인 (앱 크래시, ANR, 로딩 속도 등)
- [ ] 에러 처리 시나리오 정상 동작 확인

### 리그레션테스트 → 배포 승인 기준

- [ ] 위 필수 조건 모두 충족
- [ ] 잔여 결함에 대한 스쿼드 내 합의 완료
- [ ] QA 담당자 "QA 완료" 상태 전환

---

## 결함 우선순위 기준

| 우선순위 | 정의 | 배포 가능 여부 |
|--------|------|---------------|
| **Urgent** | 서비스 장애, 금전적 손실, 데이터 유실 | 해결 전 배포 불가 |
| **High** | 주요 기능 오동작, 우회 가능하나 사용성 저하 | 해결 전 배포 불가 |
| **Medium** | 사소한 UI/UX 이슈, 엣지 케이스 | 배포 가능, 후속 수정 |
| **Low** | 오탈자, 미세 정렬 등 | 배포 가능 |

---

## 추가 조건 (단계적 롤아웃 대상)

- [ ] 롤백 절차 확인됨
- [ ] 모니터링 지표 및 알림 설정됨
- [ ] 단계별 확대 기준 합의됨 (예: 1% → 10% → 50% → 100%)

---

## 배포 최종 확인

배포 전 QA 담당자는 다음을 확인합니다:

1. 위 종료 조건 충족 여부
2. 잔여 결함에 대한 스쿼드 내 합의
3. Linear 티켓 상태 업데이트

QA 담당자가 **"QA 완료"** 상태로 전환하면 배포 가능합니다.
